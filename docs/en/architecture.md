# Architecture Guide

This document describes the architecture that portsmith expects in generated and checked Go packages.

portsmith is a CLI-only toolkit. It does not provide runtime packages; generated code uses Gin, GORM, Chi, sqlx, and the standard library directly.

## Dependency Rule

Dependencies must point inward:

```text
Handler -> ServicePort <- Service -> RepositoryPort <- Repository
```

The arrows describe import direction. If `service.go` imports HTTP packages or `handler.go` imports database packages, the package violates the intended structure.

## Technology Stacks

portsmith supports two scaffolding/checking stacks:

- `gin-gorm`: Gin handlers and GORM repositories.
- `chi-sqlx`: Chi handlers and sqlx repositories.

Configure the stack with `portsmith.yaml`:

```yaml
stack: chi-sqlx
```

or pass `--stack` to `portsmith new` / `portsmith check`.

## Layers

### Domain Layer

Files: `domain.go`, `errors.go`

- Contains domain structs, enums, constants, and domain errors.
- Must not import HTTP routers or database clients.
- Stack-specific struct tags are allowed when needed by generated repository code.

### Interface Layer

File: `ports.go`, generated by `portsmith gen`.

- Defines minimal `XxxRepository` and `XxxService` interfaces.
- Contains only methods used by the consuming layer.
- Keeps compile-time assertions that concrete types implement the generated ports.

### Service Layer

File: `service.go`

- Implements business logic.
- Depends on repository interfaces, not concrete repositories.
- Must not import HTTP packages or SQL/database clients.

### Repository Layer

File: `repository.go`

- Implements repository interfaces.
- May use GORM, sqlx, `database/sql`, or other storage clients.
- Translates storage errors into domain-level errors.
- Must not contain HTTP logic.

### Handler Layer

Files: `handler.go`, `dto.go`, optional mapper files.

- Implements HTTP endpoints.
- Depends on service interfaces, not concrete services.
- May use Gin, Chi, `net/http`, DTOs, and request/response helpers.
- Must not import database clients.

## Wiring

Wire dependencies manually from application code:

```go
repo := orders.NewRepository(db)
svc := orders.NewService(repo)
h := orders.NewHandler(svc)
```

Constructors should accept interfaces at layer boundaries:

```go
func NewService(repo OrderRepository) *Service
func NewHandler(svc OrderService) *Handler
```

## Core Rules

| Rule id | What |
|---|---|
| `ports-required` | `ports.go` is required when handler, service, and repository files exist |
| `ports-complete` | generated ports must match the methods used by handlers/services |
| `handler-no-db` | handlers must not import database clients |
| `service-no-http` | services must not import HTTP packages or routers |
| `no-concrete-fields` | handlers/services should store interfaces, not concrete layer pointers |
| `layer-dependency` | handlers must not depend on repository types; services must not depend on handler types |
| `type-placement` | exported layer structs should live in files matching their layer |
| `file-size` | optional max lines per file |
| `cross-imports` | optional allowlist for `internal/...` cross imports |
| `constructor-injection` | constructors should accept interfaces at layer boundaries |
| `test-files` | service/handler test files should exist |
| `no-panic` | no `panic()` in service/repository files |
| `context-first` | exported service/repository methods should take `context.Context` first |
| `method-count` | optional max exported methods per service/handler |
| `wiring-isolation` | layer constructors should be called only from wiring files |
| `logger-no-other` | optional canonical logger import rule |
| `logger-no-fmt-print` | optional structured logging rule |
| `logger-no-init` | optional logger initialization rule |
| `call-pattern` | optional naming rule for `receiver.field.method()` calls |

## Configuration

```yaml
stack: chi-sqlx
lint:
  max_lines:
    - pattern: "repository*.go"
      limit: 800
  max_methods:
    - pattern: "service.go"
      limit: 25
  wiring:
    allowed_files:
      - "wire.go"
      - "app.go"
  logger:
    allowed: "log/slog"
  call_patterns:
    handler:
      allowed:
        - "*.svc.*"
      not_allowed:
        - "*.service.*"
    service:
      allowed:
        - "*.repo.*"
      not_allowed:
        - "*.repository.*"
  rules:
    test-files: { severity: warning }
```

Severity values: `error`, `warning`, `off`.

Inline suppression:

```go
//nolint:portsmith:handler-no-db
```
