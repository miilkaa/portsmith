# portsmith

**portsmith** is a Go CLI toolkit for building and maintaining Clean Architecture packages.

It provides:

- project configuration via `portsmith init`
- package scaffolding via `portsmith new`
- minimal port generation via `portsmith gen`
- mock generation via `portsmith mock`
- architecture validation via `portsmith check`

portsmith does **not** ship runtime libraries. Generated code uses normal Go ecosystem packages directly, such as Gin, GORM, Chi, sqlx, and the standard library.

## Quick Start

```bash
# Install the CLI
go install github.com/miilkaa/portsmith/cmd/portsmith@latest

# Create portsmith.yaml in an existing repo
portsmith init

# Scaffold a new package
portsmith new --stack gin-gorm internal/orders

# Implement domain, service, and repository methods, then generate ports
portsmith gen internal/orders

# Generate mocks for ports.go interfaces
go install github.com/vektra/mockery/v2@latest
portsmith mock internal/orders

# Validate architecture
portsmith check ./internal/...
```

## Clean Architecture Overview

Dependencies point inward only:

```text
Handler -> ServicePort <- Service -> RepositoryPort <- Repository
```

| File | Layer | May Know About | Must Not Know About |
|---|---|---|---|
| `domain.go` | Core | Go types | HTTP, SQL clients |
| `errors.go` | Core | standard errors | HTTP status mapping |
| `ports.go` | Interfaces | domain types | concrete HTTP/SQL implementation |
| `service.go` | Business logic | repository port, domain | HTTP, SQL |
| `repository.go` | Data access | GORM or sqlx | HTTP |
| `handler.go` | HTTP | service port, DTOs | SQL |
| `dto.go` | HTTP DTOs | request/response types | SQL |

## Project Layout Generated by `portsmith new`

```text
your-app/
├── internal/
│   └── orders/
│       ├── domain.go
│       ├── errors.go
│       ├── ports.go
│       ├── service.go
│       ├── repository.go
│       ├── handler.go
│       └── dto.go
└── main.go
```

## Technology Stacks

`portsmith new` and `portsmith check` support two stacks:

| Stack | HTTP | Database | Generated Code |
|---|---|---|---|
| `gin-gorm` | Gin | GORM | `uint` IDs, direct Gin/GORM usage |
| `chi-sqlx` | Chi v5 | sqlx + PostgreSQL | `uuid.UUID` IDs, direct Chi/sqlx usage |

Stack resolution priority:

1. `--stack gin-gorm` or `--stack chi-sqlx`
2. `stack:` in `portsmith.yaml`
3. `go.mod` heuristic: Chi module -> `chi-sqlx`, Gin module -> `gin-gorm`
4. default: `gin-gorm`

## CLI Commands

```bash
portsmith init  [--force]
portsmith new   [--stack gin-gorm|chi-sqlx] <pkg-dir>
portsmith gen   [--dry-run] [--scan-callers] [<pkg-dir|pattern>...]
portsmith mock  [<pkg-dir>...]
portsmith check [--stack gin-gorm|chi-sqlx] [<pkg-dir>...]
portsmith version
portsmith help
```

## Architecture Validation

`portsmith check` validates package-level Clean Architecture rules:

- required and complete `ports.go`
- no database imports in handlers
- no HTTP imports in services
- constructor injection through interfaces
- context-first service/repository methods
- file size and method count limits
- optional logger and call-pattern rules

Configuration lives in `portsmith.yaml`:

```yaml
stack: chi-sqlx
lint:
  rules:
    test-files: { severity: warning }
  logger:
    allowed: "log/slog"
```

See [`docs/en/architecture.md`](docs/en/architecture.md) for the full rule reference.
